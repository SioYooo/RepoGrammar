## Objective

Evaluate whether the current public-preview candidate can initialize, refresh,
and answer source-free pattern questions on three deliberately different
repository shapes without weakening `UNKNOWN`, `PARTIAL_CONTEXT`, freshness,
or metric-claim rules.

This protocol does not claim token savings. It separately checks whether the
local paired-experiment recorder is usable, but actual token savings require a
comparable baseline and treatment from real agent sessions with a documented
measurement source.

## Isolation and privacy

Each case runs from a disposable standalone clone or copy with a separate
temporary `HOME` and `.repogrammar-case-study` state directory. Commands never
use a developer's live Codex, Claude Code, agent configuration, telemetry
preference, or repository index. Temporary clones, indexes, logs, telemetry
records, and Python caches are not evidence artifacts and must not be committed.

Committed evidence is source-free. It contains public provenance, hashes,
counts, stable status tokens, low-cardinality failure classes, metric kinds,
and reason-code counts. It contains no source text, prompts, credentials,
environment values, absolute paths, raw logs, repository indexes, or telemetry
exports.

## Command matrix

Let `RG` be the product binary built from the frozen product commit, `CASE` be a
disposable case root, and `CASE_HOME` be that case's empty temporary home.
The recorded binary identified itself as `repogrammar 0.2.0-preview.0`.

```text
HOME=<CASE_HOME> REPOGRAMMAR_DIR=.repogrammar-case-study <RG> init --project <CASE> --yes --json --progress never
HOME=<CASE_HOME> REPOGRAMMAR_DIR=.repogrammar-case-study <RG> sync --project <CASE> --json --progress never
HOME=<CASE_HOME> REPOGRAMMAR_DIR=.repogrammar-case-study <RG> find --project <CASE> --json <frozen-repo-relative-target>
HOME=<CASE_HOME> REPOGRAMMAR_DIR=.repogrammar-case-study <RG> check --project <CASE> --json <frozen-repo-relative-target>
HOME=<CASE_HOME> REPOGRAMMAR_DIR=.repogrammar-case-study <RG> stats --project <CASE> --unknowns --json
```

Targets are fixed before reading query results: one application/setup module
for self-dogfood, one API-route module for the public FastAPI case, and the
single dynamic module for the negative control. Target paths are intentionally
omitted from the public summary; they remain repo-relative in the local command
ledger.

## Predeclared outcome classification

The strongest acceptable outcome is a fresh active generation followed by a
source-free supported family response. Weaker outcomes remain distinct:

- `FAMILY_MATCH`: a supported family is selected from fresh evidence;
- `PARTIAL_CONTEXT`: one target resolves and yields a read plan, but no family
  or conformance claim is supported;
- `UNKNOWN`: a typed claim is blocked, with its public reason code preserved;
- `STALE_EVIDENCE`: a `StaleEvidence` unknown or stale-index refusal; never a
  family success;
- `FALLBACK_TO_CODE_SEARCH`: repository state or an active generation is not
  available; never a query success;
- `PARSER_FAILURE`: indexing stops on a sanitized parser error; subsequent
  no-generation fallback is a consequence, not an independent product win;
- `ENVIRONMENT_FAILURE`: the command cannot reach the product oracle because
  the host lacks a required executable, permission, or network access.

Exit status is part of classification. A JSON body does not turn a nonzero
fallback or lifecycle error into success.

## Token experiment preregistration

The intended paired comparison is one frozen coding task performed twice:

1. baseline: an agent uses ordinary repository search and source reads without
   RepoGrammar output;
2. treatment: the same model/surface, task, repository commit, correctness
   tests, and stopping rule starts from RepoGrammar `find`/`check` output.

Both sessions must record complete input, output, and tool token counts from the
same accepted measurement source (`host_reported`, `user_entered`, or
`documented_tokenizer`). A pair is comparable only when both sessions complete
the same task and correctness gate. Failed correctness keeps the raw difference
diagnostic but invalidates the product claim. No tuning is allowed after seeing
either session's total.

This checkpoint ran only the recorder capability path: baseline and treatment
`record_existing` sessions were opened and stopped without
`experiment-record`, then `experiment-report` truthfully returned
`measurement_status: no_paired_measurement`, null totals, null token savings,
and unknown claim validity. Those empty records were purged. Therefore the
case-study token verdict is `NOT_MEASURED` / `no_paired_experiment`.

The telemetry help currently lists `--project` as a common option, while the
experiment parser rejects that option. The capability check therefore ran the
experiment subcommands without `--project` in an isolated home. This is a CLI
usability mismatch, not permission to invent or backfill token counts.

## Stop conditions

Stop a case immediately when:

- initialization or sync returns a deterministic parser/storage/lifecycle
  failure and one retry reproduces the same class;
- no active generation exists, in which case run `find`, `check`, and `stats`
  once only to verify truthful fallback;
- a query returns `PARTIAL_CONTEXT`, typed `UNKNOWN`, or stale evidence; record
  it without attempting to tune the repository or target into a family;
- any output would require committing source, absolute paths, raw logs,
  credentials, or unredacted telemetry;
- a real paired token measurement is unavailable.

## Validation oracle

The summary JSON must parse, all Markdown links must resolve, repository guard
must pass, root agent guides must remain byte-identical, and the committed diff
must contain only protocol/case-study/report evidence. These checks validate the
evidence artifact; they do not erase a product-runtime blocker observed by the
experiment.
