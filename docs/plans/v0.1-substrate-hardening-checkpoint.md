## Mission Contract

Harden the current v0.1 substrate before any new mining, MCP, installer, Python,
or CodeGraph runtime feature work.

The current substrate is already beyond pure repo-local lifecycle. It includes
repo-local `.repogrammar/` lifecycle, Git-aware TypeScript/JavaScript discovery,
syntax-only code-unit extraction, SQLite generation storage, active
`files`/`units` inventory reads, structural IR storage, opt-in TypeScript
semantic-worker process validation, semantic-fact storage, and internal
semantic-fact freshness/readiness gates.

That is enough to continue toward v0.1 only after hardening. Do not build family
mining, MCP serving, installer writes, Python adapters, CodeGraph integration,
or Tree-sitter replacement work in this checkpoint.

## Truth Criteria

Primary success evidence:

- the hardening fixes below are implemented against current HEAD;
- each fix has targeted positive and negative regression tests;
- specifications, plans, memories, and repository-guard docs stay consistent;
- all required validation commands pass.

Auxiliary evidence:

- current code inspection confirms whether each audit item still applies;
- tests fail before the relevant fix or directly assert the prior unsafe state;
- final report identifies any intentionally deferred release blocker.

Forbidden overclaims:

- syntax-only code units are not semantic facts or family evidence;
- fresh semantic facts are not family evidence until EC-MVFI claim builders
  exist;
- `files` and `units` inventory support is not pattern-family query support;
- successful doctor/status output is not proof that family mining is ready.

Acceptable negative outcomes:

- source-backed `NO_GO` because an audit item is already fixed on current HEAD;
- explicit blocker for a fix that requires a larger design decision;
- environment failure that cannot be safely repaired in this branch.

Do not continue to mining, MCP, installer writes, Python, or CodeGraph runtime
work while any P1 item remains unresolved.

## Required Fixes

### P1: Generation Immutability

Audit finding at `c078a82`:

- `record_ir_node`, `record_ir_edge`, and `record_semantic_fact` require the
  target generation to be `building`.
- `record_indexed_file` and `record_code_unit` open an existing generation and
  write without checking status.
- A stale `GenerationHandle` can therefore mutate validated, active, or failed
  generations through file or code-unit writes.

Implementation contract:

- Add a shared helper in the SQLite adapter, for example
  `require_building_generation(connection, generation_id, operation)`.
- Use the helper in all generation write methods:
  `record_indexed_file`, `record_code_unit`, `record_ir_node`,
  `record_ir_edge`, and `record_semantic_fact`.
- Keep error text operation-specific enough for debugging but do not leak source
  snippets or absolute paths.

Regression tests:

- `record_indexed_file` fails after validation and activation.
- `record_code_unit` fails after validation and activation.
- both writes fail for a validated-but-not-active generation.
- both writes fail for a failed generation.
- existing IR and semantic-fact status tests still pass through the shared
  helper.

### P1: Lifecycle Doctor Hygiene

Audit finding at `c078a82`:

- init ensures generated files exist but accepts existing state files without
  validating content.
- doctor/status mainly reports manifest status, missing subdirectories, and
  storage/indexing health.
- doctor should diagnose lifecycle hygiene without mutating state.

Implementation contract:

- Validate `.repogrammar/.gitignore` content and report missing or invalid
  state ignore files.
- Validate `receipts/init.json` shape and report missing or invalid receipts.
- When Git metadata exists, validate `.git/info/exclude` contains both
  `.repogrammar/` and `.repogrammar-*/`.
- Detect malformed root `.gitignore` RepoGrammar marker sections.
- Add explicit `RepositoryDoctorCode` values for the new findings, such as:
  `StateGitignoreMissing`, `StateGitignoreInvalid`,
  `GitInfoExcludeMissing`, `GitInfoExcludeIncomplete`,
  `RootGitignoreMarkerInvalid`, `InitReceiptMissing`, and
  `InitReceiptInvalid`.
- Keep `doctor` diagnostic-only. It must not recreate or repair missing entries.

Regression tests:

- corrupted `.repogrammar/.gitignore`;
- missing Git exclude file or missing required patterns;
- invalid `receipts/init.json`;
- malformed root `.gitignore` marker;
- doctor does not mutate state while reporting these findings.

### P2: Bounded Source Reads

Audit finding at `c078a82`:

- discovery and source-store paths check metadata size, then use `fs::read`,
  then check the actual byte length.
- a file can grow between metadata inspection and read, forcing an oversized
  allocation before the second check.

Implementation contract:

- Add a bounded read helper under the filesystem adapter boundary.
- Open the file and read at most `max_file_bytes + 1` bytes.
- Return `TooLarge` if more than `max_file_bytes` bytes are observed.
- Use the helper in discovery and source-store reads.
- Preserve strict hash behavior and non-UTF-8 classification.

Regression tests:

- exact max size is accepted;
- max size plus one byte is rejected;
- huge files are not read completely before the limit is enforced;
- hash mismatch behavior remains unchanged.

### P2: Parent Git Ignore Behavior

Audit finding at `c078a82`:

- `GitIgnoreChecker::new` treats a project as Git-backed only when
  `<project>/.git` exists.
- running with a project path inside a parent Git repository can miss parent
  `.gitignore` rules.

Preferred implementation contract:

- Support subdirectory projects by using
  `git -C <project> rev-parse --show-toplevel`.
- Compute each candidate path relative to the Git top-level.
- Run `git -C <git-root> check-ignore -q -- <git-relative-path>`.
- If Git is unavailable, retain the existing safe non-git fallback warning.

Regression tests:

- a project path inside a parent Git repository respects the parent
  `.gitignore`;
- non-Git roots keep deterministic fallback behavior;
- unavailable Git keeps a warning and does not crash discovery.

### P2: Manifest Validation

Audit finding at `c078a82`:

- `is_valid_bootstrap_manifest` checks JSON shape with string containment.

Implementation contract:

- Parse with `serde_json`.
- Validate:
  `schema_version == 1`,
  `state == "initialized"`,
  `storage.status` is an allowed bootstrap value,
  `indexing.status` is an allowed bootstrap value, and
  `repogrammar_version` is present and non-empty.
- Preserve compatibility with the current manifest shape.

Regression tests:

- valid JSON with different formatting and field order is accepted;
- invalid schema version, state, storage status, indexing status, or empty
  version is rejected.

### P2: Status and Doctor Schema Version Names

Audit finding at `c078a82`:

- status JSON uses `schema_version` for manifest state when storage is absent
  and storage schema version when SQLite storage is present;
- doctor JSON risks the same drift if its `checks` object exposes an ambiguous
  `schema_version` field.

Implementation contract:

- Split machine-readable status fields into `manifest_schema_version` and
  `storage_schema_version`.
- Split machine-readable doctor check fields into
  `checks.manifest_schema_version` and `checks.storage_schema_version`.
- Human output should distinguish manifest and storage schema versions.
- Update `docs/specifications/cli.md` and `docs/specifications/storage.md`.

Regression tests:

- JSON status without SQLite reports manifest schema clearly and does not emit
  ambiguous `schema_version`.
- JSON status with SQLite reports both manifest and storage schema fields when
  available.
- JSON doctor reports manifest and storage schema fields under `checks` and
  does not emit `checks.schema_version`.

### P2: Semantic Worker Request Limit

Audit finding at `c078a82`:

- Rust `MAX_WORKER_REQUEST_BYTES` is 4 KiB.
- `src/workers/typescript/worker.js` accepts 1 MiB stdin and up to 10,000
  changed files.

Implementation contract:

- Prefer aligning Rust request size to the worker's 1 MiB limit.
- Keep changed-file path validation strict.
- If a smaller Rust limit is deliberately retained, reduce and document the JS
  worker limit instead; do not leave the mismatch implicit.

Regression tests:

- many changed files below the aligned limit are accepted through Rust request
  serialization;
- over-limit requests are rejected before spawning the worker;
- path validation still rejects absolute, traversal, duplicate, and malformed
  changed-file paths.

### Release-Gate P2: Index Lock

Audit finding at `c078a82`:

- generations exist, but indexing does not yet appear to take
  `.repogrammar/locks/index.lock`.
- concurrent `index` or `sync` runs can race when choosing the next generation
  or activating the current generation.

Implementation contract:

- Implement this in the same checkpoint if it stays small and coherent.
- Otherwise, leave a source-backed release blocker in the final report and do
  not proceed to v0.1 release, MCP, installer writes, or mining until a
  dedicated lock patch lands.

Minimum lock behavior when implemented:

- create `.repogrammar/locks/index.lock` before discovery, source reads,
  generation preparation, validation, and activation;
- record pid, host when available, OS, and started timestamp;
- refuse a live lock;
- define and test stale-lock handling before deleting any lock;
- remove partial lock files if metadata writing fails;
- doctor reports lock state;
- `unlock --force --yes` may remove only a confirmed stale `index.lock`.

Current implementation status:

- `index` and `sync` acquire `.repogrammar/locks/index.lock` before discovery
  and hold it through validation and activation;
- partial metadata write failures and successful runs remove only the lock
  content they wrote;
- live or unknown lock ownership is refused, confirmed stale same-host locks
  may be replaced during acquisition, and `doctor --json` reports
  `checks.locks`;
- `unlock --force --yes` removes only a confirmed stale `index.lock`; active,
  unknown, invalid, daemon, and SQLite locks are preserved.

## Execution Model

Use a dedicated branch for the hardening series if not already on one. A name
like `feat/repo-local-lifecycle-hardening` is clearer than continuing broad
feature work under a lifecycle-only branch name.

For nontrivial execution, split work into independent lanes:

- Storage Agent: generation immutability.
- Repository Agent: lifecycle doctor/init hygiene.
- Filesystem Agent: bounded reads and parent Git ignore behavior.
- Protocol Agent: semantic worker request-size alignment.
- CLI Agent: manifest/storage schema-version reporting.
- Guard/Docs Agent: repository-guard required docs and documentation.
- Test Agent: focused regression coverage.
- Review Agent: robustness, security, and contract-drift audit.

The coordinator must integrate only reviewed changes. Preserve unrelated user
changes and other agents' edits. Do not resolve conflicts by blindly choosing a
side.

## Autonomous Round Loop

For overnight or broad autonomous execution, use at most five rounds. Each round
must:

1. preregister the specific hardening target and expected failure mode;
2. inspect current HEAD to confirm the issue still exists;
3. implement the smallest coherent fix;
4. add or update tests;
5. update canonical docs and memories;
6. run the relevant focused tests before moving to the next lane;
7. commit a coherent Conventional Commit if validation passes.

Stop early on real success, source-backed exhaustive `NO_GO`, an architecture
blocker, unsafe external action, repeated environment failure, or max rounds.

## After Hardening

Only after this checkpoint is complete should v0.1 continue toward pattern-family
evidence:

1. add a FamilyStore application/port layer over existing SQLite family tables;
2. add lightweight Express, React, Jest, and Vitest framework role facts from
   syntax-owned evidence;
3. implement EC-MVFI-lite grouping by language, code unit kind, framework role,
   and normalized shape;
4. store family records, members, variation slots, and evidence in active
   generations;
5. implement `families`, `family`, `find`, `check`, and `explain` with
   evidence-backed output or typed `UNKNOWN`;
6. add read-only MCP serving only after CLI query outputs stabilize;
7. add installer writes only after MCP self-test exists;
8. keep Python experimental and CodeGraph optional until the TS/JS family
   substrate is stable.

Do not claim semantic equivalence from syntax-only facts. Do not add LLMs,
embeddings, vector databases, or top-level graph-navigation commands.

Current post-hardening status: the FamilyStore storage substrate exists for
generation-scoped family records, members, variation slots, and family-bound
evidence. The product `index`/`sync` path runs a conservative EC-MVFI-lite
builder, but it only writes family rows when repeated compatible framework-role
candidates also have strong same-generation semantic/dataflow support. Pattern
family query commands now read FamilyStore rows when present and otherwise
return typed `UNKNOWN`. Read-only MCP serving now uses the same query preflight
and FamilyStore-backed lookup path. Narrow global Codex/Claude installer writes
now use native MCP CLIs after MCP self-test and managed receipt creation;
broader installer writes remain deferred until their ownership and receipt
contracts are specified.
