# v0.1 Parallel Development Plan

- Status: Active planning artifact
- Last updated: 2026-06-25
- Scope: Product-first v0.1 implementation coordination

This plan coordinates the next RepoGrammar v0.1 work without changing accepted
ADRs or specifications. If this file conflicts with an accepted ADR or a
specification, update the lower-priority text or write a superseding ADR.

## Product Contract

RepoGrammar is the pattern-family compression layer for coding agents. It should
return source-backed families, variation slots, exceptions, and typed `UNKNOWN`
results. It is not a generic graph-navigation product.

Official v0.1 language scope remains TypeScript and JavaScript. Experimental
Python dogfooding and optional CodeGraph provider work may happen before v0.2
only when they preserve that TS/JS product claim.

## Current Baseline

Implemented:

- repo-local lifecycle for `.repogrammar/`;
- TypeScript and JavaScript file discovery;
- dependency-free syntax-only TS/JS code-unit extraction;
- generation-scoped SQLite migrations, validation, activation, and health
  reporting;
- syntax-only `index` and `sync` wiring that stores file metadata and structural
  code-unit records;
- limited active `files`/`units` reads for syntax-only inventory/debugging;
- CodeUnit-derived structural IR node/containment-edge storage;
- semantic-fact/evidence storage substrate for validated building generations;
- internal active semantic-fact/evidence reads for future claim builders;
- internal file-hash freshness and claim-input readiness gate for active
  semantic facts, producing typed `UNKNOWN` for stale, weak, or conflicting
  facts before future claim builders can consume them;
- Rust-side TypeScript semantic-worker process/NDJSON validation;
- opt-in command-level semantic-worker fact ingestion through the
  same-generation storage gate.

Deferred:

- Tree-sitter integration;
- executable TypeScript compiler worker source;
- full repository/worktree freshness metadata and family-claim gates;
- typed IR attributes beyond the structural bootstrap graph;
- family mining and EC-MVFI scoring;
- query execution over stored family evidence;
- MCP serving;
- installer writes.

Syntax-only code units are structural candidates. They may be listed through
`files` and `units`, but they are not semantic facts, family evidence,
pattern-family query support, or conformance evidence.
Fresh semantic facts are only eligible inputs for future claim builders; they
are not family evidence until EC-MVFI compatibility, support, and contrastive
evidence checks exist.

## Phase Sequence

### Phase 1: Repo-local lifecycle

Goal: keep `.repogrammar/` lifecycle, generation activation, status, doctor,
logs, and unlock behavior stable while later writers are added.

Non-goals: semantic indexing, family mining, MCP serving, installer writes.

Agent ownership: lifecycle/storage agents own `application`, storage ports,
persistence adapters, CLI output, and storage docs. Review agents check path
safety, stale state handling, deterministic ordering, and source-snippet
exclusion.

Validation gate: full Rust quality gates, repo-guard, lifecycle/storage fixture
tests, human and JSON CLI output checks.

### Phase 1.5: Language and Provider Abstraction

Goal: define the adapter/provider boundaries that let TS/JS remain official
while experimental languages and optional graph providers stay isolated.

Non-goals: production Python support, mandatory CodeGraph dependency, provider
facts bypassing RepoGrammar-owned evidence.

Agent ownership: architecture agents own dependency rules and port contracts;
docs agents own product/spec synchronization.

Validation gate: no core dependency on parser, provider, LSP, database, MCP, or
process types; docs and ADRs stay aligned.

### Phase 1.6: Experimental Python Dogfooding Boundary

Goal: prepare an opt-in Python dogfooding boundary for FastAPI, pytest,
SQLAlchemy, and Pydantic experiments.

Non-goals: production Python support, default indexing of Python as supported
v0.1, Django, C/C++, or sound full Python semantics.

Agent ownership: language agents own future experimental adapters and fixtures;
review agents enforce claim discipline and `UNKNOWN` propagation.

Validation gate: every Python-facing output is marked experimental and cannot
upgrade syntax-only evidence into production support.

### Phase 1.7: Optional CodeGraph Provider Boundary

Goal: define a future optional lower-layer provider that can supply graph or
navigation hints without changing RepoGrammar's product identity.

Non-goals: CodeGraph dependency, CodeGraph wrapper product, top-level
CodeGraph-style CLI commands, automatic `.codegraph/` management.

Agent ownership: provider agents own optional port shape and provenance mapping;
security/review agents check freshness, conflicts, and fallback behavior.

Validation gate: RepoGrammar works when CodeGraph is absent; provider facts are
auxiliary and cannot independently prove family membership.

### Phase 1.8: UNKNOWN Governance

Goal: make typed `UNKNOWN` reasons first-class across domain, storage, CLI, MCP,
tests, metrics, and docs.

Non-goals: guessing away uncertainty, treating `UNKNOWN` as a crash, or claiming
quality gains from lower unknown rates without false-certainty controls.

Agent ownership: domain agents own taxonomy and propagation rules; test agents
own uncertain, conflicting, stale, unsupported, and dynamic fixtures.

Validation gate: structural-only facts cannot become semantic certainty;
conflicts, stale evidence, and dynamic behavior abstain with typed reasons.
Current implementation includes the first internal gate for active semantic
facts: stale evidence becomes `StaleEvidence`, weak certainty becomes
`InsufficientSupport`, and conflicting certainty becomes `ConflictingFacts`.

### Phase 2: File Discovery for Official TS/JS and Experimental Python

Goal: extend discovery policy without changing official support claims.

Non-goals: full Python semantic support or C/C++ discovery.

Validation gate: deterministic repo-relative metadata, no source snippets, no
absolute paths, correct skip reasons, and opt-in handling for experimental
Python.

### Phase 3: Storage and Generation

Goal: evolve generation storage from file/code-unit records toward IR,
semantic facts, provenance, freshness, and evidence.

Non-goals: exposing family-query claims before family-evidence read paths and
freshness semantics are tested.

Validation gate: failed generations preserve the previous active generation;
storage health is reported without implicit repair.

### Phase 4: Parsers

Goal: replace or supplement the bootstrap syntax-only extractor with
Tree-sitter-backed parser adapters that emit RepoGrammar-owned units and IR.

Non-goals: parser AST leakage into core or treating Tree-sitter as semantic
truth.

Validation gate: syntax errors produce bounded partial results and diagnostics;
all parser output carries ranges, content hashes, and provenance.

### Phase 5: Semantic and Framework Facts

Goal: execute language-native semantic workers, validate their facts against
active generations, route validated facts through the storage substrate, and add
framework adapters for TS/JS facts that syntax cannot prove.

Non-goals: TypeScript/Node compiler worker execution without version policy,
semantic-fact indexing without generation/hash matching, fabricated semantic
certainty, or unsupported TS versions producing confident facts.

Validation gate: unavailable, version-incompatible, conflicting, or dynamic
facts become `UNKNOWN` or abstention.

### Phase 6: Pattern-family Compression / EC-MVFI-lite

Goal: produce conservative family evidence using multi-view agreement,
compatibility gates, representative selection, variation slots, exceptions, and
unknowns.

Non-goals: top-k similarity only, graph navigation replacement, or unsupported
token-savings claims.

Validation gate: family claims have source-backed support and contrastive
evidence; weak evidence stays `UNKNOWN`.

### Phase 7: Query CLI and MCP

Goal: serve `find`, `family`, `explain`, `check`, richer `files`/`units`, and
the default `repogrammar_context` MCP tool from stored family evidence.

Non-goals: query claims from syntax-only generations or MCP transport that
mutates business code.

Validation gate: human and JSON outputs distinguish fallback, stale evidence,
`UNKNOWN`, and confident classifications.

### Phase 8: Install and Uninstall

Goal: implement safe agent wiring with dry-run, backups, receipts, native config
commands where available, and rollback.

Non-goals: implicit repository initialization, silent auto-update, or unmanaged
instruction-file edits.

Validation gate: malformed config is refused, receipts are reversible, and
`install`/`uninstall` never create or remove `.repogrammar/`.

### Phase 9: Release Fixtures and Smoke Gate

Goal: prove the product slice with deterministic fixtures, clean checkout
installation tests, and release smoke tests.

Non-goals: production-readiness claims without measured evidence.

Validation gate: full repository gates, installer matrix where available,
protocol fixtures, storage rollback tests, CLI/MCP smoke tests, SBOM and
artifact verification when release packaging exists.

## Parallel Agent Discipline

Use parallel agent teams for nontrivial implementation when work slices are
independent. Assign disjoint ownership by module or document set, preserve other
agents' edits, and integrate only after reviewing logic in the main session.

Common lanes:

- Context Agent: contracts, ADRs, specs, constraints.
- Implementation Agent: scoped code changes in owned modules.
- Test Agent: positive, negative, regression, stale, and unsupported cases.
- Docs Agent: canonical specs, ADRs, README, changelog, memories.
- Review Agent: false positives, false negatives, path/security issues, and
  contract drift.

## Commit Discipline

Each commit must be independently coherent, use Conventional Commits, and carry
matching tests and documentation for behavior changes. Major features stay on a
dedicated branch and merge only after required checks pass.

Planning-only commits must not include parser, storage, Python adapter,
CodeGraph integration, MCP server, mining, or installer implementation.
