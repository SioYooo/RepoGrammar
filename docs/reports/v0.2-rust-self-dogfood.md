# v0.2 Rust Structural Self-Dogfood Report

Status: implementation slice.

RepoGrammar now has a bounded Rust self-dogfood path for its own Rust
implementation. The path is intentionally structural:

- `.rs` files and `Cargo.toml` manifests are discovered by the normal
  repository-local indexer.
- `.rs` files are parsed with Tree-sitter Rust.
- Cargo manifests are read as bounded structural inventory only.
- The indexer does not run Cargo, rustc, build scripts, procedural macros, or
  project code.
- Rust facts are internal RepoGrammar evidence and do not claim general Rust
  semantic support.

Supported internal roles are limited to RepoGrammar-owned implementation
families:

- indexing phases;
- family gates;
- parser adapters;
- CLI/MCP handlers;
- installer actions;
- storage validation/recording;
- source-span/read-plan renderers;
- product tests.

Confident Rust self-dogfood family claims require compatible structural support,
fresh same-generation evidence, and no claim-relevant Rust `UNKNOWN`. Generic
Rust functions outside those RepoGrammar-owned roles remain non-family context.

Default CLI and MCP outputs remain metadata-only. Source spans are available
only through explicit source-span opt-in and the existing hash/freshness checks.

Known limitations:

- No Rust name resolution, type checking, trait resolution, borrow checking, or
  call graph.
- No Cargo feature evaluation.
- No macro/proc-macro expansion.
- No build-script execution.
- No general Rust target-language support claim.
